# Phase 2 - UI Design Contract

> Visual and interaction contract for Fluent shell rollout across Main shell, Downloader, Checker, and Logs.
> Generated by gsd-ui-researcher for planning and execution.

---

## Assumptions and Scope

No RESEARCH.md exists for Phase 2. This contract uses explicit, practical stack assumptions from the current repository:

1. UI stack remains PySide6 widgets and QSS styling.
2. Theme source of truth remains `factorio_mod_manager/ui/styles/tokens.py` plus `factorio_mod_manager/ui/styles/dark_theme.qss`.
3. Existing shell baseline in `factorio_mod_manager/ui/main_window.py` is refactored (not replaced by a web stack).
4. Existing feedback channels remain `NotificationManager` (toasts) and `StatusManager` (status rail text), with improved orchestration.
5. Phase 2 adds no backend feature scope; it restructures shell, hierarchy, and interaction quality only.

## Shell and Navigation Contract

### Primary information architecture

1. Primary navigation is a persistent left rail (D-05).
2. Page body uses a strict four-zone scaffold in all core sections (D-07):
   - Header zone
   - Primary workspace
   - Contextual side panel
   - Feedback rail
3. Global utilities stay persistently discoverable in shell header utility zone (D-08):
   - Settings
   - Help
   - Theme/appearance entry point
4. Secondary tabs are allowed only inside complex pages where needed; they do not replace the left rail (D-05).
5. Visual focal order for the shell is fixed:
   - First anchor: page heading + primary action cluster in the header zone.
   - Second anchor: active workflow/stage panel in the primary workspace.
   - Third anchor: contextual side panel details.

### Zone behavior

| Zone | Purpose | Interaction contract |
|------|---------|----------------------|
| Header zone | Title, page context, utility actions | Fixed at top of workspace; no blocking controls |
| Primary workspace | Main task surface | Owns keyboard focus order and primary CTA visibility |
| Contextual side panel | Selection details, stats, staged guidance | Collapsible; defaults open on desktop width >= 1280 |
| Feedback rail | Inline status/progress stream | Non-blocking; never steals focus |

### Downloader page structure

Downloader is a first-class page scaffold (D-06), not a simple tab refresh.

Flow is staged (D-14):
1. Parse URL
2. Review dependencies
3. Confirm download

Progressive disclosure is mandatory (D-15):
- Advanced details collapsed by default
- Core action path always visible without scrolling on 1366x768+

|-------|-------|-------|
| xs | 4px | Icon/text gaps, compact inline spacing |
| sm | 8px | Tight control spacing, row micro-gaps |

| lg | 24px | Panel/card internal padding |

| 2xl | 48px | Section breaks in tall pages |

## Motion and Feedback Contract

Motion must be subtle and purposeful (D-17).

### Timing

| Event | Duration | Easing |
|------|----------|--------|
| Surface reveal (panel/card) | 120ms | ease-out |
| Section transition (same page) | 160ms | ease-out |
| Toast enter | 140ms | ease-out |
| Toast exit | 220ms | ease-in |
| Focus ring animate-in | 80ms | linear |

### Feedback channels

1. Primary channel: inline status/progress in-page (D-09).
2. Secondary channel: toast notifications for confirmation and exception signals (D-09).
3. Toast auto-dismiss by severity (D-10):
   - success/info: 2800ms
   - warning: 4200ms
   - error: 5600ms
4. Concurrent events are queued and collapsed by semantic similarity (D-11):
   - Same event key repeated within 2s updates existing toast instead of spawning a new one.
5. Only destructive actions are blocking (D-12):
   - Delete mods
   - Clean backups

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Download Mods |
| Empty state heading | No mod selected |
| Empty state body | Paste a mod URL or search by name to load details and start a staged download. |
| Error state | We could not complete that action. Check your connection or folder settings, then try again. |
| Destructive confirmation | Delete mods: Delete selected mods? This cannot be undone. |
| Destructive confirmation | Clean backups: Delete backup folder? This cannot be undone. |

Copy tone:
- Direct and operational
- One action verb per CTA
- Error copy includes immediate next step

## Implementation Notes for Planner/Executor

1. Replace top-level `QTabWidget` shell navigation with left rail + page host while preserving existing business actions.
2. Remove inline `setStyleSheet()` usage from screen modules and move visual rules into shared QSS selectors.
3. Keep existing toast/status plumbing, but enforce queue/collapse and severity-duration rules from this contract.
4. Keep Downloader two-column workflow and make stage boundaries explicit with clear headers and progressive disclosure.
5. Ensure all core pages conform to the same four-zone scaffold before phase completion.
