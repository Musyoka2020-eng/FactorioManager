# Phase 04 - UI Design Contract

> Visual and interaction contract for Phase 04 queue control and profile workflows. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none - PySide6 widgets with custom QSS tokens and selectors (source: live codebase) |
| Preset | not applicable - desktop app, no shadcn/web preset |
| Component library | PySide6 native widgets + custom composite widgets (`QMainWindow`, `QFrame`, `QTableWidget`, `QDialog`, popup utilities) |
| Icon library | Unicode glyphs and native Qt controls only; do not add external icon dependency in this phase |
| Font | Segoe UI for app UI; Cascadia Code 9pt only for queue logs, diff technical details, and inspect views |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Queue state dots, badge interior spacing, compact inline icon gaps |
| sm | 8px | Inline action groups, chip gaps, row-level metadata spacing |
| md | 16px | Default card padding, form row gaps, modal section spacing |
| lg | 24px | Queue drawer section padding, profile card padding, dialog body padding |
| xl | 32px | Major pane gaps, diff preview gutter, empty-state padding |
| 2xl | 48px | Break between queue summary and history sections, profile library and preview zones |
| 3xl | 64px | Large empty-state vertical rhythm and full-dialog breathing room |

Exceptions: untouched legacy pre-Phase-4 host containers outside this phase may remain as-is. All new Phase 04 queue/profile layouts, embedded sub-layouts, cards, rails, and dialog internals must use only the 4/8/16/24/32/48/64 scale. Header utility buttons and icon-only controls must remain at least 32px square; the global queue badge trigger should be 36px square for reliable desktop hit area.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `BG_PRIMARY / LIGHT_BG_PRIMARY` = `#0e0e0e / #f5f5f5` | Window background, page header background, queue drawer canvas, diff preview background |
| Secondary (30%) | `BG_PANEL / LIGHT_BG_PANEL` = `#1a1a1a / #e8e8e8` | Queue item cards, profile list cards, modal panels, nav-adjacent utility containers |
| Accent (10%) | `ACCENT / LIGHT_ACCENT` = `#0078d4` | Primary CTA, active queue badge count, running-state progress, focus ring, active filter chip, selected diff scope |
| Destructive | `ERROR / LIGHT_ERROR` = `#d13438 / #d13438` | Cancel running operation, remove queued item, delete saved profile, destructive confirm emphasis only |

Accent reserved for: Apply Profile, Save Profile when it is the primary action in a modal, queue badge count and active ring, running progress bars, active navigation/focus state, selected diff filters, and inline "Open Queue" links. Do not use accent as a background for large profile cards, failed-state fills, or warning banners.

Additional semantic mapping for Phase 04: success remains `SUCCESS / LIGHT_SUCCESS` for completed items and successful undo restore; warning remains `WARNING / LIGHT_WARNING` for paused items and recoverable queue issues. Failed items use destructive red for state chips and icons, not accent blue.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Apply Profile |
| Empty state heading | Queue is clear |
| Empty state body | New downloads, updates, and profile applies will appear here. Start from Downloader or Checker. |
| Error state | This operation did not finish. Use Inspect Details, then Retry Failed Item or Skip Failed Item to keep the queue moving. |
| Apply confirmation | Review the full change list before applying this profile. Missing mods will be queued for download. |
| Destructive confirmation | Cancel Queue Item: Stop this operation now? Completed work stays as-is and remaining queued steps are canceled.<br>Delete Profile: Delete this saved profile? Installed ZIPs stay on disk. |
| Undo success | Profile applied. Undo restore is available until the next profile apply starts. |

Copy tone: direct, operational, and low-drama. Use verbs first. Prefer "Apply Profile", "Retry Failed Item", "Skip Failed Item", "Cancel Queue Item", "Pause Queue Item", "Resume Queue Item", "Inspect Details", and "Save Current as Profile". Avoid vague labels such as "Proceed", "OK", or "Continue" when a more specific action is available.

## Phase 04 Interaction Contract

### Queue Scope And Entry

- The queue is one shared operation model across downloads, checker updates, and profile apply jobs.
- There is no new left-rail destination for queue management in this phase.
- The shell header gets a persistent queue badge trigger placed in the existing utility zone between global search and settings.
- The queue trigger is always visible. When no work exists, show a neutral icon plus `0`. When work exists, show the active count in accent. When at least one failed item needs recovery, add a red state dot on the trigger.
- Queue badge trigger requirements: tooltip text `Open Queue`; accessible name `Open Queue Drawer`; accessible description must announce current queue count and whether failed items need action.
- Badge count includes `queued + running + paused + failed awaiting action`. Completed and canceled items drop out of the badge after their inline success/failure acknowledgement window ends.

### Queue Panel

- Clicking the header badge opens a non-modal queue drawer anchored under the header on the right edge of the window. It must not replace the current page.
- Drawer width: 360px minimum, 400px maximum. Height: window height minus header and 24px bottom breathing room. Keep it scrollable instead of growing wider.
- Drawer sections, in order: summary strip, active queue list, failed items, recently completed items.
- Drawer focal point: the summary strip first, then the first actionable item card. If failed items exist, the first failed card becomes the primary focal item; otherwise the running card is the focal item.
- First-read hierarchy inside the drawer: summary strip, first failed or running card, remaining active queue, then recently completed history.
- Summary strip shows total active count, one primary global control (`Pause All` or `Resume All`), and one quiet secondary control (`Clear Completed`) when applicable.
- Each queue item is a secondary-surface card with: operation title, source label (`Downloader`, `Checker`, `Profile Apply`), status chip, one progress/detail line, and context actions.
- Context actions by state:
- `queued`: drag handle, `Move Queue Item Up`, `Move Queue Item Down`, `Cancel Queue Item`
- `running`: `Pause Queue Item`, `Inspect Details`, `Cancel Queue Item`
- `paused`: `Resume Queue Item`, `Inspect Details`, `Cancel Queue Item`
- `failed`: `Retry Failed Item`, `Inspect Details`, `Skip Failed Item`
- `completed`: `Inspect Details`, `Undo Restore` only when the item is a profile apply and the snapshot is still valid
- `canceled`: `Inspect Details` only

### Inline Queue Surfaces

- Queue controls remain close to the workflow that launched them.
- Downloader gets a compact queue strip directly below `Download Mods` and above the progress console. It shows the current queued/running download item, total queued count, and an `Open Queue` action.
- Checker gets a compact queue strip between the filter bar and the table. It shows queued update/apply work and the same `Open Queue` action.
- Inline strips are summary surfaces only. Full reorder and recovery actions live in the global drawer.
- Inline status is the primary feedback channel; toasts remain secondary confirmation only.

### Queue State Model

- Required visible states: `Queued`, `Running`, `Paused`, `Completed`, `Failed`, `Canceled`.
- State chip colors: queued = secondary text/border, running = accent, paused = warning, completed = success, failed = destructive red, canceled = secondary/dimmed.
- Continue-on-failure is the default policy. A failed item never blocks later items automatically.
- Failed cards stay actionable until the user retries, skips, or clears them.
- Reordering is allowed only for items still in `Queued`. Running items are pinned to the top of the active list.

### Profiles And Presets

- Profile actions live inside the Checker workflow, not a new page.
- Checker header gains a `Profiles` entry point that opens a profile library dialog or side sheet from the existing workflow context.
- The profile library surface must expose three entry paths without nesting: `Save Current as Profile`, `Apply Profile`, `Starter Presets`.
- A profile is the desired enabled-state model of the mod set. Missing ZIPs needed to reach that state appear in the apply diff and are queued for download after confirmation.
- Starter presets are limited to three curated families in this phase: `Vanilla+`, `QoL`, `Logistics and Rail`.
- Preset cards use secondary surfaces, not accent fills. Reserve accent only for the selected preset outline and the active call to action.

### Apply Diff Preview And Confirm

- Applying a profile always opens a dedicated diff preview dialog before any queue work begins.
- Dialog width: 900px minimum. Use a two-pane structure: summary/filter rail on the left, full diff list on the right.
- Apply diff focal point: exact action counts in the left summary rail paired with the first visible diff rows on the right; users should understand the scope of change before reaching the footer CTA.
- The summary area must show exact counts for `Add`, `Remove`, `Enable`, `Disable`, and `Download`.
- The diff list must be filterable by those five action types without hiding the total counts.
- Each diff row shows mod name, action label, current state, target state, and whether a download is required.
- The footer keeps `Back` on the left and `Confirm Apply` as the only primary button on the right.
- No silent apply is allowed. Closing the dialog without confirmation leaves local state unchanged.

### Undo And Rollback Confidence

- Successful profile apply surfaces a one-click `Undo Restore` action in two places: the success toast and the queue drawer entry for the completed profile apply item.
- Undo remains available until the next profile apply begins. Do not bury undo inside Settings or a separate history page in this phase.
- Undo copy must explicitly describe that the app is restoring the pre-apply enabled-state snapshot.
- Undo never deletes locally stored ZIP files; it restores enabled-state and queued download side effects only within the retained snapshot boundary.

### Installed Mod Enable Or Disable Workflow

- Day-to-day enable or disable belongs in the installed mods table, not inside profile dialogs.
- Add a dedicated enabled-state control separate from the existing selection checkbox used for bulk actions. Do not overload the selection column.
- Disabled mods remain visible in the table with dimmed secondary text and a clear `Disabled` state treatment.
- Toggling a mod off updates `mod-list.json` only. It never deletes the local ZIP.
- Toggling a disabled mod back on should be immediate when the ZIP already exists locally; do not route simple re-enable through a download flow.
- Profile diff previews and queue entries must reuse the same enable/disable terminology as the installed-mod table.

### Blocking Versus Non-Blocking Feedback

- Blocking confirmation is required for: profile apply confirmation, canceling a running operation, and deleting a saved profile.
- `Pause Queue Item`, `Resume Queue Item`, `Retry Failed Item`, `Skip Failed Item`, and enable/disable toggles are non-blocking and should update inline state immediately.
- Toast notifications follow the existing shell rule: secondary confirmation channel only. Use action toasts only when `Undo Restore` is available.
- Inspect flows open detail dialogs or expand drawer detail in place; they must not navigate the user away from the current workflow page.

### Keyboard And Focus Behavior

- Queue badge trigger must be reachable in normal tab order from the global search field and settings button.
- `Esc` closes the queue drawer or diff preview dialog without applying changes.
- `Enter` activates the focused primary action; `Space` toggles enabled-state controls.
- Focus rings use the existing accent color. Do not create a second focus color in this phase.
