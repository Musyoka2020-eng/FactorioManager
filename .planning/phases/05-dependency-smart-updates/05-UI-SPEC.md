# Phase 05 - UI Design Contract

> Visual and interaction contract for Phase 05 dependency intelligence and smart update guidance. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none - PySide6 widgets with custom QSS tokens and selectors already established in phases 02-04 |
| Preset | not applicable - desktop app, no shadcn/web preset |
| Component library | PySide6 native widgets plus existing custom composites (`ModDetailsDialog`, `FilterSortBar`, `QueueStrip`, `QueueDrawer`) |
| Icon library | Unicode glyphs and native Qt controls only; do not add an external icon dependency in this phase |
| Font | Segoe UI for primary UI; Cascadia Code 9pt only for raw version constraints, preserved changelog code-style blocks, and technical inspect text |

Phase 05 extends the existing Fluent desktop language. No new shell destination is introduced. Dependency inspection stays inside the existing mod details popup, and smart update guidance stays inside the existing Checker and queue workflow.

## Spacing Scale

Declared values for all new Phase 05 surfaces (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | State dots, inline icon gaps, chip inner padding |
| sm | 8px | Legend gaps, compact toolbar spacing, row metadata spacing |
| md | 16px | Default card padding, panel gutters, dialog section spacing |
| lg | 24px | Dialog body padding, guidance strip padding, inspector panel spacing |
| xl | 32px | Major split-pane breathing room, tab body spacing |
| 2xl | 48px | Break between summary and deep-detail sections |
| 3xl | 64px | Large empty-state rhythm inside popup tabs and right-rail guidance states |

Exceptions:
- Existing pre-Phase-5 host containers in Checker may keep their current inherited margins until refactored.
- Icon-only controls remain at least 32px square.
- Legacy metadata rows that are preserved for backward visual continuity must normalize to the nearest approved token: use 8px for compact metadata stacks or 16px for standard stacked rows.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 10pt | 400 | 1.5 |
| Label | 9pt | 400 | 1.4 |
| Heading | 14pt | 600 | 1.2 |
| Display | 20pt | 600 | 1.2 |

Weight contract:
- Regular: 400
- Semibold: 600

Use body weight for rationale text, dependency subtitles, and changelog prose. Use label size for chips, metadata, version delta captions, and helper text. Do not introduce a third proportional weight in this phase.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `BG_PRIMARY / LIGHT_BG_PRIMARY` = `#0e0e0e / #f5f5f5` | Window background, dialog canvas, dependency tree background, changelog background |
| Secondary (30%) | `BG_PANEL / LIGHT_BG_PANEL` = `#1a1a1a / #e8e8e8` | Guidance strip, inspector cards, tab panels, right-rail detail groups |
| Accent (10%) | `ACCENT / LIGHT_ACCENT` = `#0078d4` | Primary CTA, active tab, simplified/full mode toggle, focus ring, queue-safe emphasis |
| Destructive | `ERROR / LIGHT_ERROR` = `#d13438 / #d13438` | Risky chips, incompatibility states, missing-required states, hard-stop warnings only |

Phase 05 semantic mapping:
- `Safe` = `SUCCESS / LIGHT_SUCCESS`
- `Review` = `WARNING / LIGHT_WARNING`
- `Risky` = `ERROR / LIGHT_ERROR`
- Required dependency = neutral text plus standard border, not accent-filled
- Optional dependency = secondary text and de-emphasized connector treatment
- Incompatible dependency = destructive red chip and row treatment
- Missing required dependency = destructive red chip plus broken/missing indicator
- Expansion or DLC requirement = neutral outlined chip using secondary text; do not style it like a downloadable mod and do not use accent blue

Accent is reserved for:
- `Queue Safe Updates`
- Active popup tab and simplified/full graph toggle
- Focus-visible outlines
- Queue-safe summary emphasis and inline "Why?" affordances
- Open-detail affordances that route back into the existing inspection flow

Do not use accent for Safe chips, Risky chips, changelog warning banners, or large dependency graph surfaces.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Queue Safe Updates |
| Guidance empty state heading | No update guidance yet |
| Guidance empty state body | Run Check for Updates to classify installed mods and surface safe, review, and risky recommendations. |
| Dependency empty state heading | No dependency data available |
| Dependency empty state body | This mod does not expose dependency metadata, or the portal response could not be loaded right now. |
| Changelog empty state heading | No changelog available |
| Changelog empty state body | This mod does not expose changelog entries for the selected release path. |
| Shared error state | We could not load dependency details. Check your connection, then reopen details or run Check for Updates again. |
| Safe batch helper | Only Safe updates are queued here. Review and Risky items stay manual. |
| Manual review confirm | Queue selected updates? Review and Risky items need explicit confirmation before entering the queue. |
| Expansion helper | Requires official Factorio content. This requirement is informational and cannot be queued as a mod download. |

Copy tone remains direct, operational, and low-drama. Explain why a recommendation exists in plain language. Avoid score jargon, vague labels, or version-number-only reasoning.

Destructive actions introduced in this phase: none. Manual queue confirmation for Review and Risky items is cautionary, not destructive.

## Phase 05 Interaction Contract

### Dependency Inspection Surface

- The existing `ModDetailsDialog` remains the only Phase 05 inspection popup.
- Entry points remain the existing mod-inspection paths only: Checker `View Details` and global search result details flow.
- No new left-rail item, shell page, dashboard, or detached dependency window is added.
- The dialog is expanded to a minimum size of 860x620 and a comfortable default size of 980x700 for desktop readability.
- The dialog header preserves Phase 03 structure: title, author, version metadata, installed/latest status, and description summary.
- The dialog body becomes a `QTabWidget` with fixed tab order:
  1. `Overview`
  2. `Dependencies`
  3. `Changelog`
- `Overview` remains the default tab for parity when users open details from general search or generic inspection paths.
- Executor may deep-link the popup to `Dependencies` or `Changelog` from Checker rationale affordances, but the popup itself remains the same shared surface.
- Installed-mod footer actions stay inspection-oriented. Queueing remains in Checker; do not move smart guidance primary actions into the popup footer.

### Dependencies Tab

- The `Dependencies` tab is read-only.
- The first row is a compact toolbar with:
  - `Simplified` / `Full` toggle
  - inline state legend for `Required`, `Optional`, `Incompatible`, `Missing`, and `Expansion`
  - quiet `Collapse All` control visible only in `Full` mode
- `Simplified` is the default view and shows the selected mod's direct dependency footprint first.
- `Full` expands to the complete transitive graph.
- Visualization style is a desktop-first hybrid tree inspector, not a free-pan canvas:
  - Left pane: hierarchical dependency tree (`QTreeWidget` or equivalent)
  - Right pane: selected-node inspector card
- Left pane top-level grouping is fixed in this order:
  1. `Required`
  2. `Optional`
  3. `Conflicts`
  4. `Expansion Requirements`
- Simplified mode rules:
  - Required direct dependencies are visible immediately.
  - Optional direct dependencies remain visible but visually de-emphasized.
  - Transitive branches are summarized under the direct parent with a compact child-count hint until expanded.
  - Incompatible and expansion requirements stay visible even if there are no downloadable child branches.
- Full mode rules:
  - Full transitive branches are available in the same tree.
  - Required chains expand to depth 2 by default.
  - Optional chains stay collapsed by default but remain identifiable.
  - Users can manually expand deeper levels without opening a second surface.
- Each tree row shows, in one line:
  - dependency name
  - state chip
  - version constraint if available
  - quick availability marker (`Installed`, `Missing`, `Portal`, or `Expansion`)
- The right-pane inspector shows:
  - dependency name and type
  - why this node matters in plain language
  - required-by chain from the selected root mod
  - installed status or missing status
  - whether it is queueable as a normal mod
  - expansion helper copy when applicable
- Expansion and DLC requirements are displayed as non-downloadable requirements only. They never expose queue, update, or download actions.
- Incompatible nodes are visually separated from normal dependencies and use destructive styling.
- Missing required dependencies use destructive styling even in Simplified mode.

### Changelog Tab

- The `Changelog` tab lives inside the same `ModDetailsDialog`.
- When installed version is known, the tab opens with a pinned delta summary card at the top:
  - installed version
  - latest version
  - count of relevant entries between those versions when available
- The first changelog section is `Changes since your installed version` and expands the newest relevant entries first.
- Older history appears below in collapsed sections or expandable version rows.
- When installed version is unknown or the dialog was opened from a portal-only result, show the latest entry expanded first and place older history below it.
- Preserve line breaks and list structure from portal content. Do not over-summarize or rewrite the changelog body.
- Long changelog bodies scroll inside the tab body only; the dialog header and footer remain stable.
- If no changelog data exists, show the changelog empty state rather than an error treatment.

### Checker Smart Guidance Workflow

- Smart update guidance lives in the existing Checker workflow and feeds the existing queue controller.
- The current `Update All` batch action is replaced by `Queue Safe Updates`.
- `Update Selected` remains the manual queue path for selected rows, including Review and Risky items after explicit confirmation.
- No bypass path is introduced. All safe, review, and risky queue actions continue through the shared queue workflow used in Phase 04.

### Checker Layout Changes

- Keep the existing three-column Checker page scaffold from Phase 02 and Phase 04.
- Add a new full-width `SmartUpdateStrip` between the existing queue strip and the mod table.
- Add a `Guidance` column immediately after the existing `Status` column in the installed mods table.
- Extend the existing filter bar with a guidance filter combo:
  - `Any guidance`
  - `Safe`
  - `Review`
  - `Risky`
- Extend the existing right sidebar by adding a `Selected Update Guidance` group beneath `Statistics`.

### Checker Visual Hierarchy

- First-read focal point is the `SmartUpdateStrip` summary and its `Queue Safe Updates` action.
- Second focal point is the installed-mod table, specifically outdated rows and the new `Guidance` column that explain triage at scan speed.
- Third focal point is the `Selected Update Guidance` panel, which expands rationale and offers the bridge into the shared details popup.
- The queue strip remains visible above the smart guidance strip but stays secondary unless active queue work already exists.

### SmartUpdateStrip Contract

- The strip summarizes the active scope in one sentence plus count chips:
  - `X Safe`
  - `Y Review`
  - `Z Risky`
- Scope rule is fixed:
  - if one or more rows are selected, the strip summarizes the current selection
  - if no rows are selected, the strip summarizes currently visible outdated rows after active filters
- The strip's primary accent CTA is `Queue Safe Updates`.
- `Queue Safe Updates` queues only rows classified `Safe` within the current scope.
- Review and Risky rows are never included in the one-click safe batch, even if they are selected.
- If the current scope contains no Safe rows, disable the primary CTA and show helper text explaining why.
- The strip includes a quiet `Why not all?` affordance that reveals a short explanation: safe batch is conservative by design and Review or Risky items remain manual.
- The strip is informational and actionable; it does not replace the global queue strip.

### Guidance Column Contract

- Every outdated row receives exactly one guidance chip in the new `Guidance` column:
  - `Safe`
  - `Review`
  - `Risky`
- Up-to-date rows show no guidance chip.
- Chip colors follow the semantic mapping in this contract and do not use accent blue.
- Hover or selection of a row must expose rationale in the right sidebar, not in a tooltip-only interaction.
- Table sorting by guidance should order `Risky`, `Review`, `Safe`, then empty rows when sorting descending for triage.

### Selected Update Guidance Panel

- The right sidebar gains a second group box titled `Selected Update Guidance`.
- One selected row:
  - show the guidance chip
  - show 2-4 plain-language rationale bullets
  - show dependency delta summary (`no dependency change`, `adds optional branch`, `adds required dependency`, `conflicts with installed mod`, etc.)
  - show changelog availability status
  - show `View Details` button that opens the shared popup and may deep-link to `Dependencies` or `Changelog`
- Multiple selected rows:
  - show aggregated Safe/Review/Risky counts
  - show reminder that only Safe items enter the one-click batch
  - show `Open Queue` as the quiet action if queue items already exist
- No selection:
  - show the guidance empty state for this panel
- Rationale text must be plain language and specific. Do not show opaque scores or unexplained risk levels.

### Manual Queue Confirmation Contract

- `Update Selected` continues to use the existing queue workflow, but behavior changes by selection mix:
  - Safe-only selection: queue immediately through the shared queue path
  - Selection containing any Review or Risky items: open a lightweight confirmation sheet before queueing
- The confirmation sheet summarizes selected counts by class and repeats the manual review confirmation copy.
- Footer actions for the confirmation sheet are fixed:
  - `Queue Selected`
  - `View Details`
  - `Return to Checker`
- `View Details` opens the existing popup for the primary selected mod; it does not open a new risk dashboard.

### Queue Integration Rules

- `Queue Safe Updates` produces a normal queue-backed update operation and updates the existing Checker queue strip and global queue drawer.
- Manual `Update Selected` uses the same queue operation type.
- Review and Risky items remain manually queueable only through these existing queue-backed actions.
- No direct execution or bypass button is allowed inside the guidance strip, guidance panel, dependency tab, or changelog tab.

### Keyboard And Focus Behavior

- Focus order on Checker remains: header actions, left workflow buttons, filter bar, queue strip, smart update strip, table, right sidebar.
- `Enter` on the focused `Queue Safe Updates` button queues only Safe rows in scope.
- `Enter` on a selected row or `Ctrl+I` may open `View Details` if a shortcut is added, but it must route into the same shared popup.
- `Esc` closes the popup or manual confirmation sheet without queueing anything.
- Focus rings continue using the existing accent color only.
